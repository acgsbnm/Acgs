Security and Operational Requirements New in v1.6.1
Transport and network posture
  • HTTPS is mandatory in every environment. The minimum supported transport version is TLS 1.2. Plain HTTP is not supported.
  • Supported deployment modes are internet-facing (default), VPN-restricted, or IP-allowlisted. In VPN or allowlist mode, disallowed source networks may receive 403 FORBIDDEN before normal request processing.
  • Use separate base URLs, credentials, monitoring, and alerting per environment. Production and non-production traffic must not share keys.
Auth lifecycle, throttling, and replay protection
  • X-Integration-Key is issued per environment and integration partner. Keys must be rotatable without downtime, with controlled overlap during planned cutovers. Raw keys must never be written to logs or support tickets.
  • Rate limits may vary by environment. Clients must treat 429 and transient 5xx responses as retryable, honor Retry-After when present, and use bounded exponential backoff.
  • Every write request must include a unique Idempotency-Key. Retries after ambiguous failures must reuse the same key. Reusing a key with a different request body must return 409 DUPLICATE_IDEMPOTENCY_KEY.
PII handling and image fetch controls
  • Customer, address, VAT, payment-reference, and gift-card recipient data are sensitive. Mask or redact these fields in logs, monitoring, and non-production test data. Sensitive card data is never exposed by this API.
  • When imageUrl is used, the source must be a public HTTPS CDN or object-storage URL. Private/internal hosts, localhost, link-local addresses, RFC1918 ranges, and cloud metadata endpoints are not supported and must be rejected.
  • Image fetch policy: maximum 3 redirects, 15-second download timeout, 10 MB max size after download or base64 decode, and accepted formats limited to JPEG, PNG, GIF, and WebP. Return 413 for oversized payloads, 415 for unsupported formats, and 422 INVALID_IMAGE_SOURCE for blocked or invalid remote sources.
Content Model and Write Semantics New in v1.6.1
Single-language catalog contract
  • This version of the contract is intentionally single-language only. Product name, descriptions, and meta fields represent a single live storefront value, not a translation set.
  • Request and response payloads do not include localized translation arrays. GET /api/integration/v1/reference/languages remains a lookup for store defaults and optional UrlRecord.LanguageId usage only.
Omit vs null vs clear
  • PUT /items/by-sku/{sku}: required core fields must always be supplied. For existing products, omitted optional scalar fields remain unchanged. For new products, omitted optional scalar fields fall back to nop defaults.
  • PATCH /items/by-sku/{sku}: omitted fields remain unchanged. Use this endpoint for partial updates, unpublish actions, and soft deletes.
  • PUT /items/by-sku/{sku}/full-sync: on existing products, omitted child objects and collections remain unchanged. Explicit empty arrays on categories, tags, specifications, attributes, images, and combinations mean clear and replace that collection. Clear explicit store mappings by sending { "allStores": true }.
  • Use null only to clear optional scalar, date, or text fields. Sending null for required or non-nullable fields, or for related collections and mapped entities, must return 422 VALIDATION_ERROR.
  • Empty string may be used to clear optional free-text fields such as admin comments and meta text. SEO slug behavior remains explicit through PUT /items/by-sku/{sku}/seo-slug.
SKU Identity Rules New in v1.6.1
Parent product routes
  • Endpoints shaped like GET, PUT, or PATCH /api/integration/v1/items/by-sku/{sku} treat {sku} as the parent or base product SKU.
  • For a product such as BAS3000 with combinations like BAS3000-GN-XL, call product routes with BAS3000, not the child SKU.
  • This same parent-SKU rule applies to related product routes such as /attributes, /combinations, /images, /seo-slug, /full-sync, and base pricing routes such as /pricing/by-sku/{sku}.
Child combination SKUs
  • Variant or combination SKUs belong inside the request or response body as variantSku or combinationSku, not in the parent product route.
  • Use /items/by-sku/{sku}/combinations or /items/by-sku/{sku}/full-sync to create and maintain child combinations under the parent product.
  • Use child SKUs directly on inventory endpoints only. Those endpoints auto-detect whether the SKU belongs to a base product or a ProductAttributeCombination.
  • Variant-specific price overrides and stock belong on the combination record, not on the parent product route. Use combinations[].overriddenPrice and variant stock updates for child SKUs.
