Required headers (all endpoints): X-Integration-Key (auth), X-Correlation-Id (optional; server generates if absent), X-Store-Id or ?storeId= (optional; defaults to current store). Write endpoints also require: Idempotency-Key.
Security and Operational Requirements
Transport and access
  • HTTPS is required in every environment.
  • Use separate credentials and monitoring per environment.
  • Share restricted integration credentials rather than broad administrative access.
Retries and idempotency
  • All write operations require an Idempotency-Key.
  • Retry transient failures with bounded exponential backoff.
  • Reusing an idempotency key with a different payload returns 409 DUPLICATE_IDEMPOTENCY_KEY.
Sensitive data handling
  • Customer, address, VAT, payment-reference, and recipient fields should be treated as sensitive.
  • Sensitive card data is never exposed by this interface.
  • When image URLs are used, they must be public HTTPS resources.
Content Model and Write Semantics
Catalog content model
  • This contract assumes a single live storefront value for names, descriptions, and meta text.
  • Localized translation arrays are not part of this extract.
Omit vs null vs clear
  • PUT /items/by-sku/{sku} is the full base-item upsert contract for the external item record.
  • PATCH /items/by-sku/{sku} updates only the fields supplied in the request body.
  • Use null only where the target field is defined as clearable; omit fields that should remain unchanged.
SKU Identity Rules
Parent product routes
  • Item and pricing routes use the parent or base product SKU.
  • Base item routes should not be called with child or combination SKUs.
Child and stock-specific routes
  • Inventory endpoints may be configured to resolve either a base SKU or a stock-bearing child SKU.
  • Where RFQ or return flows are feature-gated per site, confirm enablement through GET /meta/capabilities before implementation.
